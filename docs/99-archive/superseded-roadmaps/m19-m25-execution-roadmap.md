# M19-M25 里程碑任务计划

## 1. 阶段目标

M19-M25 的核心目标是把系统从“核心模块可用”推进到“团队可长期试用和迭代”的协同平台形态。

推进顺序：

1. M19 先稳定 IM，因为 IM 是入口、通知承载面和跨模块分享入口。
2. M20 建立通知中心，把项目、文档、表格、审批、IM 事件汇聚到用户可感知入口。
3. M21 强化文档协作，让文档具备评论、提及、版本、权限和 IM 分享能力。
4. M22 推进多维表格 MVP，形成 Airtable/飞书多维表格的轻量骨架。
5. M23 强化项目与 BUG 管理，让研发团队能日常使用。
6. M24 收口权限、审计、搜索治理，降低后续模块扩展风险。
7. M25 做交付、部署、备份、监控和发布流程，让系统可持续运行。

## 2. 总览

| 里程碑 | 名称 | 目标 | 主要风险 | 完成标准 |
| --- | --- | --- | --- | --- |
| M19 | IM 多端可靠性与体验收敛 | 稳定消息、会话、已读、重连、重试和多端刷新 | 消息重复、断线丢消息、未读不一致 | IM 核心链路可稳定内测 |
| M20 | 通知中心与全局事件入口 | 建立站内通知和统一未读入口 | 通知泛滥、重复通知、跳转不准确 | 用户能从通知跳到具体对象 |
| M21 | 文档协作 MVP 强化 | 文档进入真实协作流 | 编辑体验复杂、权限和评论定位不一致 | 文档可评论、提及、分享、搜索 |
| M22 | 多维表格 Base MVP | 表、字段、记录、视图可用 | 字段类型扩散、前端表格复杂度上升 | 能创建表、字段、记录并分享 |
| M23 | 项目与 BUG 管理增强 | 支撑研发团队日常需求、任务、BUG 管理 | 工作流过度设计、状态不可控 | BUG 生命周期和看板可用 |
| M24 | 权限、审计、搜索统一治理 | 平台底座收口 | 权限散落、搜索泄露、审计缺失 | 权限、审计、搜索有统一测试矩阵 |
| M25 | 交付与运维 | 可部署、可备份、可恢复、可监控 | 环境不可复现、数据不可恢复 | 本地/测试环境可重复交付 |

## 3. M19 IM 多端可靠性与体验收敛

目标：让 IM 成为稳定核心入口，为通知、文档、表格、项目的协作事件承载打基础。

| 编号 | 任务 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| M19-T01 | 梳理 IM 当前能力、缺口、风险清单 | IM 风险清单和执行文档 | 覆盖发送、重连、已读、成员、消息操作、多端契约 |
| M19-T02 | 消息发送状态 | 前端发送中、失败、重试状态；后端发送幂等 | 同一 `clientMessageId` 重试不重复落库 |
| M19-T03 | WebSocket 重连与断线恢复 | 重连刷新策略；`messageSeq` 同步游标 | 重连后能补齐历史，不依赖客户端时间 |
| M19-T04 | 多端会话排序、未读、已读一致性 | 已读游标校验和多端刷新事件 | 非当前会话消息不能作为 read marker |
| M19-T05 | 成员变更实时同步 | 成员增删、退出、关闭会话事件推送 | 多端会话详情和列表能刷新 |
| M19-T06 | 消息操作回归 | 编辑、撤回、置顶、表情事件类型准确化 | 操作后所有成员端刷新一致 |
| M19-T07 | 多端 API 契约整理 | 消息响应包含 `messageSeq`、`clientMessageId` | Web、桌面、移动可复用发送和补拉契约 |
| M19-T08 | 自动化测试与质量门禁 | IM 集成测试和质量门禁报告 | `pnpm verify` 通过 |

依赖：M15 实时通道基础、M18 多端基础。

当前状态：已推进第一轮，见 `docs/m19-im-reliability-plan.md`。

## 4. M20 通知中心与全局事件入口

目标：让用户能通过一个统一入口看到需要处理或关注的事项，并能跳转到具体消息、文档、事项、表格记录或审批。

| 编号 | 任务 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| M20-T01 | 通知领域模型设计 | `notifications` 模型补强或迁移 | 支持 recipient、type、title、body、target、read 状态、dedupeKey |
| M20-T02 | 通知写入服务 | `NotificationService` | 支持幂等创建、批量创建、按用户查询 |
| M20-T03 | 通知列表 API | `/api/notifications` 查询、分页、筛选 | 可按未读、类型、时间分页 |
| M20-T04 | 通知已读能力 | 单条已读、批量已读、全部已读 | 未读数实时或准实时更新 |
| M20-T05 | 前端通知中心页面 | 通知列表、未读角标、空状态 | 点击通知能打开目标对象 |
| M20-T06 | IM 通知接入 | 提及、群成员变更、重要消息通知 | 不重复通知本人发出的普通消息 |
| M20-T07 | 项目/文档/表格事件接入 | 事项分配、评论、文档提及、表格记录提及 | 通知内容可读、目标跳转准确 |
| M20-T08 | 通知测试矩阵 | 服务、API、前端基本回归 | 覆盖去重、权限、已读、跳转 |

关键约束：

- 通知不能直接替代业务事件，通知是事件的用户可见投影。
- 通知目标必须使用平台对象模型和 deep link。
- 不能向无权限用户创建可泄露标题或正文的通知。

完成标准：

- 用户能在侧边栏或顶部看到未读通知数。
- 通知能跳转到 IM 消息、项目事项、文档、表格记录。
- `pnpm verify` 通过，通知相关测试覆盖去重和已读。

## 5. M21 文档协作 MVP 强化

目标：让文档不只是可编辑文本，而是能参与团队协作、评论、提及、分享和搜索。

| 编号 | 任务 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| M21-T01 | 文档信息架构补强 | 文档目录、最近访问、收藏入口 | 用户能快速找到常用文档 |
| M21-T02 | Block 编辑体验优化 | 新增、编辑、删除、排序块 | 基础块操作稳定，移动端不破版 |
| M21-T03 | 文档评论模型 | 文档级和块级评论 API | 评论可创建、解决、删除 |
| M21-T04 | 文档提及成员 | `@成员` 解析和通知 | 被提及成员收到通知并可跳转 |
| M21-T05 | 文档版本/变更记录 | 版本记录、变更摘要、恢复入口 | 可查看最近版本并恢复 |
| M21-T06 | 文档权限接入强化 | 查看、编辑、管理、分享权限 | 无权限用户不可读标题/正文 |
| M21-T07 | IM 分享文档卡片增强 | 文档卡片包含标题、摘要、权限态 | IM 粘贴文档链接显示正确卡片 |
| M21-T08 | 文档测试补强 | API、权限、评论、通知测试 | 覆盖文档协作主路径 |

关键约束：

- 不做复杂多人实时协同编辑，先做可靠的块编辑和评论协作。
- 文档卡片、搜索结果、通知目标必须走平台对象摘要。
- 文档权限必须和搜索权限一致。

完成标准：

- 用户能创建文档、编辑块、评论、提及、分享。
- 文档能被 IM、通知、搜索三条链路正确引用。
- 文档权限测试通过。

## 6. M22 多维表格 Base MVP

目标：做出轻量多维表格骨架，覆盖表、字段、记录、视图、筛选排序和协作引用。

| 编号 | 任务 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| M22-T01 | Base 领域模型确认 | Base/Table/Field/Record/View 模型 | 模型支持后续字段类型扩展 |
| M22-T02 | 字段类型一期 | 文本、数字、日期、成员、单选、多选 | 字段校验和存储稳定 |
| M22-T03 | 表格视图 API | 记录分页、字段配置、视图配置 | 支持按视图读取记录 |
| M22-T04 | 前端表格视图 | 表格渲染、单元格编辑、加载状态 | 常规数据量下可用 |
| M22-T05 | 记录详情页 | 单条记录查看、编辑、评论预留 | 可从通知/IM/搜索打开记录 |
| M22-T06 | 筛选和排序 | 视图级筛选、排序条件 | 条件可保存到视图 |
| M22-T07 | IM 分享表格/记录卡片 | Base/Table/Record 对象摘要 | 粘贴链接显示卡片并可跳转 |
| M22-T08 | Base 权限和搜索接入 | 权限过滤、索引更新、测试 | 搜索不泄露无权限记录 |

关键约束：

- 字段类型先少后多，不在 MVP 做公式、自动化、复杂关联。
- 记录值存储必须保留结构化能力，避免后续字段扩展困难。
- 表格视图要考虑移动端只读或轻编辑体验。

完成标准：

- 能创建 Base、表、字段、记录。
- 能用表格视图编辑常用字段。
- 表格对象能被 IM 分享、搜索、通知引用。

## 7. M23 项目与 BUG 管理增强

目标：让项目模块支撑研发团队的需求、任务、BUG 日常管理。

| 编号 | 任务 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| M23-T01 | 事项类型增强 | 需求、任务、BUG 类型和字段差异 | 不同类型展示不同必要字段 |
| M23-T02 | 状态流转与工作流 | 状态配置、流转校验 | BUG 生命周期可控 |
| M23-T03 | 负责人、优先级、截止日期 | 字段、筛选、排序 | 可按负责人和优先级管理 |
| M23-T04 | 看板视图 | 状态列、拖拽流转、列表兜底 | 看板和列表状态一致 |
| M23-T05 | 事项评论与附件 | 评论、附件、活动记录 | 讨论和文件沉淀到事项 |
| M23-T06 | 跨对象关联 | 关联文档、IM 消息、表格记录 | 事项详情能查看关联对象 |
| M23-T07 | BUG 验证记录 | 复现步骤、修复版本、验证人、验证结果 | BUG 关闭前有验证依据 |
| M23-T08 | 项目模块回归测试 | API、权限、看板、关联测试 | 项目主路径稳定 |

关键约束：

- 研发团队优先，先覆盖内部研发流程，不做通用项目管理大而全。
- BUG 管理要有验证闭环，不只是一个 issue 类型。
- 事项和 IM、文档、表格必须能互相引用。

完成标准：

- 团队能用项目模块管理需求、任务、BUG。
- BUG 能从新建、处理中、已修复、待验证、关闭完成闭环。
- 事项详情能关联 IM 消息和文档。

## 8. M24 权限、审计、搜索统一治理

目标：把平台底座收口，避免继续扩展模块时出现权限散落、搜索泄露、审计缺失。

| 编号 | 任务 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| M24-T01 | 权限模型盘点 | 模块权限矩阵 | 明确每类对象的 view/edit/manage/share/delete |
| M24-T02 | 统一权限服务收口 | `PermissionDecisionService` 扩展和规范 | 新模块不直接散写权限判断 |
| M24-T03 | 搜索权限过滤强化 | 搜索查询统一调用权限服务 | 无权限对象不返回敏感字段 |
| M24-T04 | 搜索索引重建能力 | 索引重建脚本/API/管理入口 | 可从数据库重建搜索索引 |
| M24-T05 | 审计日志查询 API | 按用户、对象、动作、时间查询 | 管理员可查关键操作 |
| M24-T06 | 管理后台审计页面 | 审计列表、筛选、详情 | 可定位谁在何时做了什么 |
| M24-T07 | 敏感操作审计补齐 | 登录、权限、删除、导出、成员变更审计 | 关键操作都有 audit log |
| M24-T08 | 权限与审计测试矩阵 | 自动化测试和检查清单 | 覆盖越权、搜索泄露、审计缺失 |

关键约束：

- 权限、搜索、对象卡片、通知必须共享一致的可见性判断。
- 审计日志不能记录明文密码、token、敏感密钥。
- 搜索索引重建不能破坏在线数据。

完成标准：

- 搜索不返回无权限对象。
- 关键操作有审计记录并可查询。
- 权限判断集中且有测试覆盖。

## 9. M25 交付与运维

目标：让系统从开发态进入可持续使用状态，具备重复部署、备份恢复、监控和发布检查能力。

| 编号 | 任务 | 交付物 | 验收标准 |
| --- | --- | --- | --- |
| M25-T01 | Docker Compose 一键启动整理 | 本地依赖和应用启动说明 | 新环境可按文档启动 |
| M25-T02 | 初始化账号和测试数据脚本规范化 | 初始化、清理、演示数据脚本 | 数据脚本可重复执行 |
| M25-T03 | 数据备份脚本 | Postgres、MinIO 备份脚本 | 可生成带时间戳备份 |
| M25-T04 | 数据恢复演练 | 恢复脚本和演练记录 | 能从备份恢复可登录环境 |
| M25-T05 | 日志目录和轮转规范 | 后端、前端、脚本日志规范 | 日志不污染仓库，不无限增长 |
| M25-T06 | 健康检查与监控指标 | Actuator、Prometheus、基础告警 | 能判断服务、数据库、对象存储健康 |
| M25-T07 | 生产/测试部署草案 | 环境变量、反向代理、TLS、存储规划 | 可部署到测试服务器 |
| M25-T08 | 发布检查清单 | 发布前检查、回滚、备份、版本记录 | 每次发布有固定流程 |

关键约束：

- 本地开发、测试环境、生产环境配置必须分层，不把本地密钥提交到仓库。
- 备份恢复必须演练，不只写脚本。
- 发布流程要和 `pnpm verify`、数据库迁移、数据备份绑定。

完成标准：

- 新机器可按文档启动依赖和应用。
- 数据可备份、可恢复。
- 发布前有明确检查清单和回滚步骤。

## 11. 每轮工作循环

每轮必须执行：

1. 明确本轮覆盖的任务编号，例如 `M20-T01 到 M20-T04`。
2. 读取当前代码和历史文档，确认已有实现，不重复造轮子。
3. 输出或更新对应里程碑文档。
4. 小步实现后端、前端、迁移和测试。
5. 运行定向测试。
6. 运行 `pnpm verify`。
7. 汇总完成项、风险、测试结果和下一步。

提交前必须确认：

- 不提交 `.local-logs/`、`.local-reports/`、构建产物。
- 不提交与本轮无关的脚本或文档，除非用户明确要求。
- 数据库重置只在用户明确要求时执行。
