# PILOT-V2 合成试运行与产品化工程收口路线

## 1. 路线目标

本路线不再扩张新的大功能域，而是把当前 V049、知识库唯一模型、用户工作台与管理后台双 UI 的完成态，收口为可由 5 个合成人格连续、重复、故障注入验证，可回归、可运维、可判断工程 Go/No-Go 的协同平台版本。真实用户体验和采用意愿不在本路线内宣称完成，后续进入真实小范围试用前仍需补充人工证据。

最终必须回答四个问题：

1. 当前数据、权限、搜索和对象引用是否可信。
2. IM、项目、知识内容、Base、审批、通知、搜索的关键流程是否可稳定重复运行。
3. 合成普通成员和管理员是否能在各自 UI 中按脚本完成任务，并保持预期权限边界。
4. 发生故障、错误授权或试运行失败时，是否能定位、备份、回退和形成明确决策。

## 2. 当前基线

- 数据库基线为 Flyway V001-V049。
- 知识内容唯一模型为 `KnowledgeBaseSpace`、`KnowledgeBaseItem`、`KnowledgeContent` 和 `knowledge_content`。
- 用户工作台与管理后台双 UI v1 已完成，仍共享一个 SPA 和后端进程。
- 最近一次路线级全量验证为后端 60 tests passed、V001-V047 空库迁移和前端 lint/build 通过；本路线后续目标验证已把当前 schema 推进至 V049，新的路线级全量验证留在 M11。
- M1 已完成历史悬空知识引用和旧 `/docs/...` 路径的备份后清理，当前一致性基线无该例外。
- 旧 M31/M40 数据、旧 `/docs` 协同 E2E 和知识库 v1/v2 迁移脚本已经归档，不得作为本路线基线。
- 远程仓库当前没有生效中的 CI 合并门禁；治理文档、活动脚本和运维脚本纳入版本管理，`.local-*` 运行产物、日志、备份、环境文件和密钥不提交远程。

## 3. 不做范围

- 不在本路线实现 Yjs CRDT、多节点 Redis pub/sub 或多节点协同部署。
- 不在本路线实现 Base 公式、自动化、字段级权限和记录级权限。
- 不建设正式 Electron 安装包、原生移动端、Kubernetes 或自动扩缩容。
- 不恢复独立文档模块、`/docs`、`/api/docs`、`Document*` 或 `document` 产品 objectType。
- 不以增加页面数量代替试运行问题闭环。

## 4. 执行与验证规则

1. 每次 AI 工作循环只推进一个里程碑内的连续任务，不跨里程碑合并收口。
2. 中间 checkpoint 使用 `light`；涉及后端行为时，里程碑 finish 使用 `stage` 和目标测试集合。
3. 完整后端测试、完整集成测试和 Flyway 空库全量迁移只在 PILOT-V2-M11 路线收口执行；安全或迁移高风险变更可提前升级验证。
4. 前端行为变更必须执行对应模块浏览器冒烟；不再运行归档 M31/M40 脚本。
5. 数据修复必须先生成预览和备份，禁止为了让检查通过直接删除或改写共享数据。
6. 每个里程碑更新本文件状态，并生成 `docs/90-reports/pilot-v2-mX-execution-report.md`。

#### M10 指标口径

| 指标 | 计算方式 | 通过阈值 | 说明 |
| --- | --- | --- | --- |
| `personaAuthenticationRate` | 成功通过真实登录页的人格数 / 5 | `= 100%` | 不等同真实活跃率 |
| `scenarioAttemptRate` | 有完整原始记录的场景执行数 / 计划执行数 | `= 100%` | 失败场景也必须留记录 |
| `scenarioSuccessRate` | 首次执行成功的场景数 / 实际执行数 | `>= 95%` | 重跑不能覆盖首次失败 |
| `moduleRoundCoverage` | 每轮至少成功一次的模块数 / 6 | 每轮 `= 100%` | IM、项目、知识、Base、审批、搜索 |
| `unexpectedAuthorizationCount` | 实际允许/拒绝与矩阵不一致的次数 | `= 0` | 任意信息泄漏触发停止 |
| `dataConsistencyViolationCount` | 重复对象、状态倒退、悬空引用、搜索/通知不一致总数 | `= 0` | 由一致性检查和场景断言复算 |
| `openCriticalIssueCount` | 轮次结束未关闭的 P0/P1 数 | `= 0` | P0 立即停止，P1 必须有明确决定 |
| `faultRecoveryRate` | 在约定时限内恢复并续跑成功的注入故障数 / 注入故障数 | `= 100%` | 只统计预先声明的受控故障 |
| `automationFlakeRate` | 无产品状态变化但同输入结果不一致的脚本次数 / 总执行数 | `<= 5%` | 单独归类，不混入产品成功率 |

明确排除 `satisfactionScore`、真实活跃率、学习成本和采用意愿；这些指标只能由真实参与者产生。

## 7. 全局验收标准

- 数据：无未解释的悬空引用、旧产品路径、孤立权限和搜索泄漏。
- 可靠性：核心流程可由隔离 fixture 重复执行，失败能留下截图、trace、请求和数据标识。
- 权限：成员、部门、用户组、角色授权结果与解释一致；无权限用户不获取对象敏感信息。
- 用户体验：普通成员通过用户工作台完成任务，管理能力只从管理入口出现。
- 管理体验：管理员通过管理后台完成成员生命周期、组织、角色、风险和审计治理。
- 运维：备份、恢复、健康检查、发布和回退至少完成一次桌面或隔离环境演练。
- 合成试运行：5 个合成人格完成至少 3 个独立轮次，六模块覆盖、故障恢复和指标阈值通过，无未关闭 P0/P1。
- 人工证据边界：不得用合成试运行推断满意度、学习成本、自然协作效率或采用意愿；进入真实使用前必须单独补人工验证。
- 路线收口：PILOT-V2-M11 执行 `route-final`，完整测试、迁移、前端和关键浏览器回归通过。

## 8. 路线之后的决策分支

工程 Go 后先进入真实小范围试用；只有补齐人工证据后，才根据试运行证据选择下一条主路线，不并行铺开：

1. 如果内容协同冲突和多节点需求最突出，进入知识内容 CRDT/多节点协同路线。
2. 如果结构化业务数据需求最突出，进入 Base 公式、自动化和细粒度权限路线。
3. 如果移动和桌面使用阻塞最突出，进入正式多端交付路线。
4. 如果可靠性和运维仍是主要问题，继续产品化收口，不扩展功能面。
