# 轻量协同工作平台里程碑与执行任务路线图

## 1. 文档目的

本文档承接当前已完成的 PRD、技术架构、Lark 产品形态分析、技术选型和项目骨架初始化成果，将最终目标拆解为可执行里程碑和具体任务。

当前最终目标不是做一个简单 IM，也不是一次性复刻飞书，而是建设一个面向内部研发团队的多端协同工作空间：

- IM 是核心入口、通知入口和业务对象路由器。
- 项目/Bug 是研发协作主流程。
- 文档和多维表格承载知识沉淀和结构化数据协作。
- Web、桌面端、移动端共享账号、权限、对象链接、通知和 API。
- 技术上采用 Spring Boot 3.5.x 模块化单体 + React Web + pnpm workspace，预留后续模块拆分。

## 3. 执行原则

### 3.1 里程碑定义

每个里程碑必须满足：

- 有可运行交付物。
- 有端到端验收路径。
- 有测试或验证命令。
- 不依赖口头说明才能判断是否完成。
- 不把深水区一次性做完，先交付闭环，再增强。

### 3.2 优先级规则

- P0：没有它无法形成主流程闭环。
- P1：第一阶段应做，但可在 P0 闭环后补齐。
- P2：增强能力，进入第二阶段。

### 3.3 建议节奏

建议按两周一个 Sprint 推进。每个里程碑可包含 1 到 3 个 Sprint，具体取决于团队人数。

每个 Sprint 固定输出：

- 本轮任务清单。
- 数据库迁移。
- 后端接口。
- 前端页面或交互。
- 基础测试。
- 可演示路径。
- 已知风险和下一轮入口。

- 使用初始管理员账号可登录。

- 被禁用成员无法登录。
- `GET /api/auth/me` 返回当前用户、角色、设备信息。
- 所有业务 API 默认要求认证，`/api/health` 除外。

## 17. 推荐执行顺序

第一阶段 P0 推荐按以下顺序推进：

2. M2 平台对象、事件、通知与文件底座。
3. M3 IM MVP。
4. M4 项目与 Bug MVP。
5. M5 文档 MVP。
6. M6 多维表格 MVP。

## 18. 第一阶段最小可用版本定义

第一阶段完成后，团队应能完成以下真实工作流：

1. 管理员创建成员。
2. 成员登录平台。
3. 成员进入 IM 沟通。
4. 在项目群中讨论问题。
5. 从讨论中创建 Bug 或任务。
6. 在项目看板中推进状态。
7. 在 Bug 评论中 @成员。
8. 被 @成员在 IM 或通知中心收到提醒。
9. 创建文档沉淀需求或技术方案。
10. 将文档关联到任务或 Bug。
11. 创建多维表格管理需求池或测试用例。
12. 在 IM 中分享 Bug、文档、表格记录并显示权限安全的卡片。
13. 工作台展示未读、待办和最近内容。

## 19. 风险与控制

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 功能边界扩张 | 第一阶段延期 | 每个里程碑只做闭环，不做完整飞书能力 |
| IM 可靠性不足 | 用户信任下降 | 消息先落库再推送，断线补拉，客户端去重 |
| 权限散落 | 越权风险 | 所有读取和卡片都经过 PermissionService |
| 多维表格模型过重 | 开发和性能风险 | 第一阶段限制字段类型和记录规模 |
| 文档协同过早复杂化 | 研发成本失控 | 第一阶段不用 CRDT，用版本冲突检测 |
| 多端同时开发 | 主线分散 | Web 优先，桌面/移动先做协议和轻量入口 |
| 运维后置 | 上线质量不足 | M11 持续穿插，至少 CI、备份、审计先做 |

## 20. 下一步建议

下一步建议进入 M1，并以 Sprint 1 的形式执行：

- 完成管理员初始化。
- 完成登录、JWT、refresh token、退出登录。
- 完成 `CurrentUser` 和 Spring Security 鉴权。
- 完成前端登录态和路由守卫。
- 完成成员管理最小闭环。

Sprint 1 结束时必须能演示：

- 管理员登录。
- 新增成员。
- 成员登录。
- 未登录无法访问业务页。
- 后端测试和前端构建通过。
