# 轻量协同工作平台 PRD + 技术架构草案

## 1. 背景与目标

本项目面向内部研发团队，建设一个以 IM 为核心入口的多端协同工作空间，集成项目管理、Bug 管理、文档、多维表格和后续 OA 流程。

这里的“轻量”不是功能简陋，而是相比 Lark/飞书这类大型平台，优先服务团队内部真实协作流程，避免过早覆盖通用企业办公场景。

### 1.1 产品目标

- 提供团队日常沟通、任务协作、文档沉淀、项目推进的一体化入口。
- 让 IM 成为工作流触发中心，而不是独立聊天工具。
- 让项目、Bug、需求、文档、表格之间可以互相引用和通知。
- 让核心组件在 Web、桌面端、移动端共享同一套账号、权限、链接、通知和对象模型。
- 第一阶段优先满足研发团队内部使用，不追求 SaaS 化、多租户和复杂生态。
- 技术上采用模块化单体，预留后续按模块拆分微服务的空间。

### 1.2 非目标

第一阶段不做以下能力：

- 不做完整飞书/钉钉/企业微信替代品。
- 不做音视频会议。
- 不做复杂企业通讯录和大型组织管理。
- 不做完整在线 Office 套件。
- 第一阶段不交付完整移动端 App，但平台协议、对象模型、通知模型必须支持后续移动端接入。
- 不做外部客户协作空间。
- 不做复杂低代码平台。
- 不做多租户 SaaS 计费、套餐、商业化能力。

## 2. 目标用户与使用场景

### 2.1 用户角色

| 角色 | 说明 | 核心诉求 |
| --- | --- | --- |
| 团队成员 | 研发、测试、产品、设计等内部成员 | 沟通、查看任务、编辑文档、处理 Bug |
| 项目负责人 | 项目 owner、技术负责人、产品负责人 | 管理进度、分配任务、跟踪风险 |
| 测试人员 | 负责 Bug 提报、验证、回归 | 管理缺陷生命周期 |
| 管理员 | 系统配置与权限维护人员 | 成员管理、权限配置、空间管理 |
| 后续 OA 审批人 | 部门负责人或流程审批人 | 处理请假、报销、采购等审批 |

### 2.2 高频使用场景

- 团队成员进入平台后，优先看到 IM 会话和通知。
- 项目群中讨论需求、Bug、任务，并可直接创建或引用项目事项。
- 在 Bug 评论中 @相关人员，被 @人员在 IM 中收到通知。
- 项目负责人通过看板查看任务状态、阻塞项和延期项。
- 团队用文档沉淀需求、设计方案、会议纪要和复盘。
- 团队用多维表格管理清单类数据，例如需求池、版本计划、测试用例、资产清单。
- 后续通过 OA 模块处理请假、报销、采购、权限申请等流程。

## 3. 产品定位

本平台定位为“研发团队内部协同工作台”，核心不是把多个工具简单堆在一起，而是围绕研发协作建立统一对象模型：

- 人：成员、角色、权限。
- 消息：沟通、通知、提醒。
- 事项：需求、任务、Bug、审批。
- 内容：文档、表格、附件。
- 空间：项目、群组、知识空间。

IM 是入口，项目管理是主工作流，文档和多维表格是知识与结构化数据载体。平台整体形态不是“Web 平台 + 多端 IM”，而是“多端工作空间 + IM 核心入口”。

核心组件都应具备跨端访问能力：

- Web：完整生产力端。
- 桌面端：接近 Web 的常驻工作端。
- 移动端：消息、通知、查看、评论、轻量处理和后续审批。

### 5.1 账号与组织

- 用户资料。
- 团队成员列表。
- 邀请成员。

- 权限优先采用 RBAC，后续在文档、项目、表格上叠加资源级权限。

### 5.2 IM 模块

IM 是核心入口，承担沟通、通知、工作流触发三类职责。IM 必须支持 Web、桌面端和移动端接入，第一阶段 Web 优先交付，但后端协议、消息模型、未读同步、通知和鉴权不能绑定单一 Web 客户端。

#### 会话类型

- 单聊。
- 群聊。
- 项目群。
- 系统通知会话。

#### 客户端形态

- Web 端：第一阶段主交付形态。
- 桌面端：基于 Web 技术封装，后续提供 Windows/macOS/Linux 客户端。
- 移动端：后续提供 iOS/Android 客户端，要求支持系统推送、会话列表、消息收发、通知和业务卡片查看。

#### 消息类型

- 文本消息。
- 图片消息。
- 文件消息。
- 内部链接卡片。
- 系统通知。
- 业务对象引用卡片，例如需求、任务、Bug、文档、表格空间、数据表、表格记录。

#### 基础功能

- 会话列表。
- 消息收发。
- 未读数。
- @成员。
- 消息已读状态。
- 历史消息分页加载。
- 文件上传与下载。
- 从消息创建任务或 Bug。
- 在消息中引用任务、Bug、文档、表格。
- 发送内部业务链接后自动生成缩略卡片。
- 多客户端消息同步、已读同步和未读数一致。

#### 后置功能

- 消息撤回。
- 消息编辑。
- 表情回应。
- 置顶消息。
- 群公告。
- 移动端完整项目/文档/表格编辑能力。
- 外部网页链接预览。
- 复杂机器人。
- 音视频。

## 7. 技术架构草案

### 7.1 架构原则

- 模块化单体优先。
- 数据库模型保持清晰，避免过早拆库。
- 实时能力独立封装，预留后续拆分 IM 服务。
- 通知、权限、搜索作为横向基础模块。
- 每个业务模块通过领域服务和事件机制解耦。

### 7.2 推荐技术栈

#### 前端

- React。
- TypeScript。
- Vite。
- Ant Design 或 shadcn/ui。
- TanStack Query。
- Zustand 或 Redux Toolkit。
- TipTap 或 Milkdown 用于文档编辑。
- AG Grid、TanStack Table 或自研轻量表格组件用于多维表格。

#### 后端

推荐优先选择：

- Spring Boot + Java/Kotlin。

备选：

- NestJS + TypeScript。

如果团队 Java/Spring 更熟，建议选 Spring Boot。该平台包含 IM、权限、工作流、复杂数据模型和长期维护，Java 生态在企业协作类系统上更稳。

#### 数据与基础设施

- PostgreSQL：主数据库。
- Redis：缓存、会话、WebSocket 状态、消息队列轻量场景。
- MinIO：本地对象存储。
- Meilisearch 或 Elasticsearch：第二阶段引入全文搜索。
- Docker Compose：本地开发和小规模部署。

### 7.3 后端模块划分

建议按模块化单体组织：

- `identity`：用户、登录、组织、成员。
- `permission`：角色、权限、资源授权。
- `im`：会话、消息、已读、在线状态。
- `project`：项目、任务、Bug、迭代。
- `doc`：文档、版本、评论。
- `base`：多维表格、字段、记录、视图。
- `notification`：通知、事件投递。
- `file`：附件、对象存储。
- `search`：索引、检索。
- `workflow`：后续 OA 流程。
- `audit`：审计日志。

### 7.4 前端模块划分

- `auth`：登录和会话。
- `layout`：全局布局。
- `messenger`：IM。
- `projects`：项目管理。
- `issues`：任务/Bug 详情。
- `docs`：文档空间和编辑器。
- `tables`：多维表格。
- `notifications`：通知中心。
- `admin`：管理后台。
- `shared`：通用组件、API client、hooks、权限工具。

## 9. 关键技术设计

### 9.1 IM 实时通信

第一阶段采用 WebSocket。

核心流程：

1. 用户登录后建立 WebSocket 连接。
2. 服务端维护用户连接状态。
3. 发送消息时先写入数据库。
4. 写入成功后发布消息事件。
5. 在线用户通过 WebSocket 收到消息。
6. 离线用户通过未读数和通知补偿。

关键要求：

- 消息必须先持久化再推送。
- 客户端发送消息需要本地临时 ID，服务端返回正式 ID。
- WebSocket 断线后需要拉取 missed messages。
- 群聊大规模扩散后续再优化，第一阶段可直接按成员推送。

### 9.2 文档协同

推荐分阶段实现。

第一阶段：

- 文档内容存 JSON 或 Markdown。
- 保存时检查版本号。
- 版本冲突时提示用户合并。
- 每次保存生成版本快照。

第二阶段：

- 支持评论、局部引用和段落级定位。
- 支持历史版本对比。

第三阶段：

- 引入 Yjs/CRDT 实现实时协同。
- 协同服务可从单体中拆出。

### 9.3 多维表格

多维表格的核心难点不是 UI，而是数据模型、字段类型、权限和筛选查询。

第一阶段建议：

- 每个表使用统一记录表。
- 字段定义保存在 `base_fields`。
- 单元格值保存在 `base_record_values`。
- 根据字段类型做服务端校验。
- 常用字段建立冗余索引列或 JSONB 索引。

需要谨慎处理：

- 字段类型变更。
- 删除字段后的历史数据。
- 成员字段与权限联动。
- 附件字段的生命周期。
- 大表分页和筛选性能。

### 9.4 权限

权限判断分三层：

1. 系统级角色。
2. 空间/项目级角色。
3. 文档、表格、事项等资源级权限。

建议所有写操作都经过统一权限服务判断，不在各模块中散落硬编码。

### 9.5 通知事件

业务模块不直接投递消息，而是发布领域事件。

示例事件：

- `IssueAssigned`
- `IssueCommentMentioned`
- `DocumentMentioned`
- `MessageMentioned`
- `TableRecordAssigned`
- `ApprovalRequested`

通知模块订阅事件后生成站内通知或 IM 系统消息。

### 9.6 搜索

第一阶段：

- PostgreSQL 模糊查询和简单全文搜索。

第二阶段：

- 引入 Meilisearch 或 Elasticsearch。
- 索引对象包括消息、项目事项、文档、表格记录。

搜索权限必须后置过滤，避免越权访问搜索结果。

## 10. API 设计原则

- REST API 起步，WebSocket 处理实时事件。
- 接口按业务资源设计。
- 所有列表接口支持分页。
- 所有写接口返回最新资源状态。
- 所有关键业务对象包含 `createdBy`、`createdAt`、`updatedBy`、`updatedAt`。
- 状态变更接口单独建模，避免通用 update 绕过业务规则。

示例：

- `POST /api/auth/login`
- `GET /api/conversations`
- `GET /api/conversations/{id}/messages`
- `POST /api/conversations/{id}/messages`
- `GET /api/projects`
- `POST /api/projects/{id}/issues`
- `POST /api/issues/{id}/transition`
- `GET /api/docs`
- `PUT /api/docs/{id}/content`
- `GET /api/bases/{id}/tables/{tableId}/records`
- `POST /api/bases/{id}/tables/{tableId}/records`

## 12. 主要风险

### 12.1 功能边界失控

风险：

- IM、文档、表格、项目、OA 都有大量深水区。

控制方式：

- 每阶段只完成一个闭环。
- 复杂功能明确后置。
- 不以飞书完整功能作为第一阶段验收标准。

### 12.2 IM 可靠性

风险：

- 消息丢失、重复、乱序、未读不准。

控制方式：

- 消息先落库再推送。
- 客户端使用消息 ID 去重。
- WebSocket 断线后补拉。
- 未读数从服务端计算。

### 12.3 文档协同复杂度

风险：

- 实时协同编辑涉及 CRDT/OT、冲突合并、光标同步、离线恢复。

控制方式：

- 第一阶段不做实时协同。
- 用版本号和冲突提示控制复杂度。
- 后续再引入 Yjs。

### 12.4 多维表格性能与模型复杂度

风险：

- 动态字段、筛选、排序、权限会导致查询复杂。

控制方式：

- 第一阶段限制单表规模。
- 常用字段建立索引策略。
- 字段类型设计保持克制。

### 12.5 权限复杂度

风险：

- 资源间引用后容易出现越权访问。

控制方式：

- 所有资源读取前统一鉴权。
- 搜索结果做权限过滤。
- 关键操作写审计日志。

## 13. 第一阶段验收标准

第一阶段完成后，团队应该可以：

- 登录平台并看到工作台。
- 在 IM 中进行单聊和群聊。
- 在项目群中讨论并创建任务/Bug。
- 在看板中推进任务/Bug 状态。
- 对任务/Bug 评论、@成员、上传附件。
- 创建和编辑文档。
- 创建基础多维表格并维护记录。
- 收到与自己相关的通知。
- 管理员可以维护成员和基础权限。
