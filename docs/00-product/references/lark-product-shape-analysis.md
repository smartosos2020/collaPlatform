# Lark 产品形态分析与本平台调整建议

> 参考资料：本文记录阶段性产品形态判断，不是当前功能清单。涉及“文档”独立模块、客户端范围和后续建议的内容可能已被知识库唯一入口及双 UI 路线替代。

## 1. 分析目的

本平台不应只把 IM 做成多端可用，也不应把文档、项目、表格、OA 做成几个松散模块。参考 Lark/飞书的产品形态，更合理的方向是：

- 多端统一工作空间。
- IM 作为核心入口、通知中心和业务对象路由器。
- 文档、表格、项目、审批等组件共享账号、权限、搜索、通知、链接、文件和对象模型。
- 不同端承载不同深度的编辑能力，但都能查看、流转、评论、通知和跳转。

## 2. Lark 的公开产品形态观察

根据 Lark 官方站点和产品页面，Lark 将自身定位为 productivity superapp，核心产品包括 Messenger、Docs、Base、Meetings、Minutes、Calendar、Email、Approval、Wiki、OKR、Open Platform 等。Lark 下载页也提供 MacOS、iOS、Windows、APK、Linux 等客户端。

这说明 Lark 的形态不是“IM + 插件”，而是“统一工作空间 + 多应用组件 + 多端客户端”。

## 3. Lark 产品组织方式

### 3.1 统一工作空间

Lark 的用户不是先进入某个孤立应用，而是进入一个工作空间。工作空间里有消息、文档、知识库、审批、日历、会议、多维表格等入口。

对我们的启发：

- 平台入口不应只有 IM 页面。
- 应有统一工作台、统一导航、统一搜索、统一通知。
- 各模块对象必须能相互引用。

### 3.2 Messenger 是入口，不是全部

Messenger 的价值在于：

- 触达用户最快。
- 承载通知。
- 承载讨论上下文。
- 承载业务对象卡片。
- 从会话中进入文档、任务、表格、审批。

对我们的启发：

- IM 是核心入口和事件流，不是唯一主界面。
- 项目、文档、表格仍要有完整模块页面。
- IM 中分享任何业务对象都应生成可鉴权卡片。
- 所有业务事件都应能回到 IM 或通知中心。

### 3.3 组件跨端可用

Lark 的产品不是只有 Messenger 多端，Docs、Wiki、Approval、Base 等也出现在统一产品体系中。不同端的能力深度不同，但基本访问、查看、评论、通知、审批、跳转能力需要跨端成立。

对我们的启发：

- Web：完整生产力端，承担复杂编辑和配置。
- 桌面端：接近 Web 能力，强调常驻、通知、快速入口。
- 移动端：强调消息、通知、查看、评论、轻量处理，不强求复杂表格/文档编辑第一阶段完整。

### 3.4 对象链接和权限是平台底座

Lark 的 Docs、Wiki、Base、Approval 等组件能组织在一起，前提是对象链接、权限、通知和搜索是平台底座，而不是每个应用各做一套。

对我们的启发：

- 需求、Bug、文档、表格记录、审批单都应是平台对象。
- 每个对象有统一的类型、ID、标题、摘要、权限、链接、更新时间。
- IM、搜索、通知、工作台都使用这套对象能力。

### 3.5 移动端不是简单 WebView

协同平台移动端的核心不是把桌面页面缩小，而是处理：

- 系统推送。
- 离线补拉。
- 快速查看。
- 快速评论。
- 快速审批。
- 业务卡片跳转。
- 文件预览。

对我们的启发：

- 移动端后续应独立设计，优先 IM、通知、任务、审批、文档查看。
- 多维表格复杂配置、文档复杂编辑、项目看板管理可以优先留给 Web/桌面端。

## 4. 本平台产品形态调整

### 4.1 从“IM 多端”调整为“平台多端”

原先表述：

- IM 需要支持 Web、桌面、移动端。

调整后：

- 平台核心组件都需要多端可访问。
- IM 是第一入口和事件流。
- Web 是完整生产力端。
- 桌面端是 Web 能力的常驻增强。
- 移动端是消息、通知、查看、评论、轻量处理端。

### 4.2 组件跨端能力分级

| 模块 | Web | 桌面端 | 移动端 |
| --- | --- | --- | --- |
| IM | 完整 | 完整 | 完整 |
| 工作台 | 完整 | 完整 | 精简 |
| 通知中心 | 完整 | 完整 | 完整 |
| 项目/Bug | 完整 | 完整 | 查看、评论、状态流转 |
| 文档 | 完整编辑 | 完整编辑 | 查看、评论、轻量编辑 |
| 多维表格 | 完整配置和编辑 | 完整配置和编辑 | 查看、筛选、轻量编辑 |
| 文件 | 上传、下载、预览 | 上传、下载、预览 | 预览、上传图片/文件 |
| OA | 后续完整 | 后续完整 | 后续审批优先 |
| 管理后台 | 完整 | 完整 | 不优先 |

### 4.3 平台对象模型

需要抽象统一的平台对象：

- `issue`
- `document`
- `base`
- `base_table`
- `base_record`
- `file`
- `notification`
- `approval_instance`

统一对象能力：

- 获取对象摘要。
- 获取对象权限。
- 生成内部链接。
- 生成 IM 卡片。
- 生成通知目标。
- 进入对应客户端路由。

### 4.4 统一路由和 Deep Link

所有端应共享一套内部链接语义：

```text
colla://issue/{id}
colla://doc/{id}
colla://base/{baseId}/table/{tableId}
colla://base-record/{recordId}
https://colla.local/issues/{id}
https://colla.local/docs/{id}
```

Web 打开 HTTPS 链接。

桌面端可注册 `colla://` 协议。

移动端可通过 universal link 或 app link 打开。

### 4.5 统一通知模型

通知不应只为 IM 服务，而应服务整个平台。

通知目标可以是：

- 消息。
- 任务/Bug。
- 文档。
- 表格记录。
- 审批单。

通知投递渠道：

- Web/桌面 WebSocket。
- IM 系统会话。
- 移动推送。
- 后续邮件或企业微信。

## 5. 对当前设计的具体调整

### 5.1 产品层

- 将“IM 多客户端接入”升级为“平台多客户端接入”。
- 第一阶段仍 Web 优先，但平台对象、权限、链接、通知、API 不绑定 Web。
- 移动端第一阶段不交付完整 App，但移动端能力分层需要在设计中明确。

### 5.2 技术层

- 设备模型不只服务 IM，而是服务全平台。
- `user_devices`、`push_tokens` 放在 identity/notification 基础能力中。
- 增加 `platform_objects` 或对象摘要服务，不一定第一阶段建实体表，但需要有统一接口。
- 所有业务对象提供 summary/card resolver。
- 前端路由和后端链接解析都基于统一对象类型。

### 5.3 工程层

- `web/` 是完整生产力端。
- `desktop/` 复用 Web 端。
- `mobile/` 后续独立 React Native/Expo。
- 共享 API contract 和对象类型定义，避免 Web、桌面、移动端各自理解业务对象。

## 6. 第一阶段建议落点

第一阶段不需要同时开发三端完整产品，但必须做到：

- API 不绑定 Web。
- 登录支持设备登记。
- 通知模型支持多渠道。
- 所有核心对象有统一链接。
- IM 卡片使用统一对象摘要。
- Web 页面至少移动浏览器可读，复杂编辑可降级。
- 桌面端和移动端目录预留，但不抢占 MVP 主线。

## 7. 结论

本平台应定位为“多端协同工作空间”，而不是“Web 协同平台 + 多端 IM”。

IM 的地位是：

- 第一入口。
- 通知入口。
- 讨论入口。
- 业务对象卡片入口。
- 工作流触发入口。

其他组件的地位是：

- 项目/Bug 是研发工作流主线。
- 文档是知识和方案沉淀。
- 多维表格是结构化协作数据。
- OA 是后续流程能力。

多端策略是：

- Web/桌面端承担完整生产力。
- 移动端承担触达、查看、评论、轻量处理和审批。
- 所有端共享对象、权限、通知、链接和 API。
